# PCIe Transaction Layer Rule Pack

Rule pack: `pcie-tlp`

## Rule basis

- Require Completion Status evidence for all outstanding Non-Posted requests.
- Treat non-Successful-Completion (non-SC) without explanation as a hard stop.
- Treat Poisoned TLPs as a hard stop requiring AER correspondence.
- Require CfgRd without a matching CplD to be flagged as a timeout.
- Treat memory requests to unassigned BAR space as a hard stop.
- Treat Tag reuse before Completion as a hard stop (routing ambiguity).
- Require TLP Fmt/Type and Length consistency detectable via CATC decode.
- Warn when Traffic Class fields do not map to an established VC.
- Warn on incorrect Byte Enable combinations for multi-DWORD requests.

## Hard-stop rules

- `PCIE5-TLP-001`: Completion with non-SC status (UR, CA, CRS) must be explicitly explained.
  An unexplained UR/CA Completion is a hard stop.
- `PCIE5-TLP-002`: Poisoned TLP (EP bit=1 in TLP header) must have a corresponding AER
  Uncorrectable Error record. Poisoned TLP without AER evidence is a hard stop.
- `PCIE5-TLP-003`: Memory request (MRd or MWr) to an address range not assigned via BAR
  sizing is a hard stop (rogue access or software addressing error).
- `PCIE5-TLP-008` (hard stop): Tag value reuse — same 8-bit/10-bit Tag in two concurrent
  Non-Posted requests (CfgRd, MRd, IO Read/Write) before the first Completion is returned
  causes completion routing ambiguity. This is a hard stop requiring requester investigation.
- `PCIE5-TLP-013` (hard stop): TLP Length field (bits[9:0] of DW1) must match actual
  payload DWORDs. CATC-detected length mismatch (header says N DWORDs, payload has M ≠ N)
  is always a hard stop; this is a Malformed TLP.

## Warn rules \u2014 Completion handling

- `PCIE5-TLP-004` (warn): CfgRd without a corresponding CplD within the capture window
  suggests a Completion Timeout. Check Device Status Completion Timeout bit and AER log.
- `PCIE5-TLP-005` (warn): CRS (Configuration Retry Status) appearing more than once for
  the same request suggests the device is not completing initialization on time.
- `PCIE5-TLP-006` (warn): TLP header Fmt/Type field combination (Fmt[2:0]:Type[4:0]) not
  matching a valid PCIe encoding is a Malformed TLP; warn and cross-reference AER Malformed TLP bit.

## Warn rules \u2014 Byte Enables

- `PCIE5-TLP-007` (warn): For requests with Length > 1 DW, Last DW BE must be non-zero
  (cannot be 0x0 except for 1-DW requests or special aligned accesses). Invalid Last DW BE
  causes Malformed TLP at the completer; warn for BAD BE combination.

## Warn rules \u2014 Addressing and routing

- `PCIE5-TLP-009` (warn): Message TLP routing field (bits[2:0] in Message TLP header Type)
  must match the intended delivery:
  - 000b: Routed to Root Complex
  - 001b: Broadcast from Root Complex
  - 010b: Routed by Address
  - 011b: Routed by ID (B/D/F)
  - 100b: Implicit (local termination)
  Incorrect routing type for a given Message Code is a protocol violation.
- `PCIE5-TLP-010` (warn): Traffic Class (TC) field in any TLP header (bits[6:4] of DW0)
  must correspond to an enabled Virtual Channel (VC) in the VC Extended Capability. TC with
  no VC mapping causes TLP to be silently dropped by switches; warn.
- `PCIE5-TLP-011` (warn): Type 1 Configuration Request (CfgRd1/CfgWr1, targeting a non-zero
  Bus Number) must only be issued by a Root Port or Upstream Switch Port. Endpoint generating
  Type 1 Config requests warns for incorrect device enumeration assumptions.

## Warn rules \u2014 Tag management

- `PCIE5-TLP-012` (warn): Extended Tag (8-bit or 10-bit) must only be used when Extended Tag
  Field Enable is set in Device Control (bit8=1) and the completer confirms Extended Tag support.
  Sending 8-bit tags without Enable warns for possible requester non-compliance.

## Review prompts \u2014 Completion status

- Were any non-SC Completions (UR, CA, CRS) observed? What is the explanation per request?
- Was Poisoned TLP present? Was AER Uncorrectable Error logged?
- Which CfgRd or MRd had no matching CplD? What was the register/address being read?

## Review prompts \u2014 TLP format and addressing

- Were all MRd/MWr addresses within assigned BAR ranges?
- Was any Tag value reused before its Completion was returned?
- Were Byte Enables valid for the request Length (Last DW BE non-zero for multi-DW)?
- Were all Traffic Class values within the range of established VCs?

## Review prompts \u2014 Message TLPs

- Were Message TLPs (PM, ERR_FATAL, PME_TO_Ack, etc.) routed correctly?
- Were Type 1 Config requests only generated by the Root Port?
- Were Extended Tags used appropriately (Enable bit set, completer supports it)?

## Hard-stop rules

- `PCIE5-TLP-001`: Completion with non-SC status (UR, CA, CRS) must be explicitly explained.
  An unexplained UR/CA Completion is a hard stop.
- `PCIE5-TLP-002`: Poisoned TLP (EP bit set in TLP header) must have a corresponding AER
  Uncorrectable Error record. Poisoned TLP without AER evidence is a hard stop.
- `PCIE5-TLP-003`: Memory request (MRd or MWr) to an address range that was not assigned
  via BAR sizing is a hard stop (rogue access or software bug).

## Warn rules

- `PCIE5-TLP-004` (warn): CfgRd without a corresponding CplD within the capture window
  suggests a Completion Timeout; warn and check AER Completion Timeout status.
- `PCIE5-TLP-005` (warn): Completion RetryStatus (CRS) observed more than once suggests the
  device is not ready; warn if device was expected to be functional at that point.
- `PCIE5-TLP-006` (warn): TLP Length field inconsistent with the actual payload size
  (detectable via CATC decode error) warrants warn for protocol compliance review.

## Review prompts

- Were any non-SC Completions (UR, CA, CRS) observed? What were the causes?
- Were Poisoned TLPs present? Was there a corresponding AER record?
- Did any CfgRd or MRd lack a matching Completion within the capture?
- Were all Memory requests confined to the assigned BAR address ranges?
- Was CRS observed, and was device readiness confirmed afterward?
